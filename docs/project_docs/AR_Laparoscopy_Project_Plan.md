## 2. МОДУЛЬ 1 — UI-КЛИЕНТ (Unity)

### 2.1 Назначение
Главное клиентское приложение для Android-планшета. Отображает AR-сцену, принимает 3D-модели с сервера, реализует оба режима работы (Федорцов / Коновалов), все инструменты измерений и UX-потоки.

### 2.2 Стек и библиотеки

| Компонент | Решение | Версия / примечание |
|---|---|---|
| Движок | Unity | 2022.3 LTS (минимум) |
| AR-ядро | ARFoundation | ≥ 5.x, поверх ARCore |
| ARCore | Google ARCore | ≥ 1.35.0 |
| Компьютерное зрение | OpenCV for Unity | Распознавание ArUco-маркеров |
| Язык | C# | Весь клиентский код |
| UI-фреймворк | Unity UI (Canvas) | Адаптация под планшет, 48×48 dp кнопки |
| Загрузка моделей | Unity WebRequest | HTTP GET → binary STL → GameObject |
| 3D-модели | Unity Mesh API | Прямая загрузка бинарного STL в Mesh |
| Кэш | Unity FileCache (custom LRU) | До 2 ГБ, очистка по 30 дням |

| Компонент | Решение | Примечание |

| Веб-фреймворк | FastAPI | Async, OpenAPI из коробки |
| DICOM-обработка | pydicom | Чтение тегов, валидация серий |

| Серверная ОС | Linux | CPU ≥ 8 ядер, RAM ≥ 32 ГБ |
| Хранилище файлов | Локальная FS (SSD) | /data/stl/, /data/dicom/ |

### 3.3 Test API vs Production API — концепция

Ключевой вопрос из контекста: *как тестировать без доступа к Амире (реальной сегментации)?*

**Решение — единый FastAPI с режимами:**

```
POST /api/v1/dicom/upload          ← загрузить DICOM-архив (ZIP)
POST /api/v1/dicom/segment/{id}    ← запустить сегментацию
GET  /api/v1/models/{patient_id}   ← получить STL + JSON
GET  /api/v1/patients              ← список пациентов

Режим TEST:
  └─ сегментация = TotalSegmentator (локально, без GPU на CPU ~5 мин)
     или pre-generated STL из mock_data/ (мгновенно)

Режим PROD:
  └─ сегментация = вызов внешней Амиры через gRPC/REST
     (подставляется через переменную окружения SEGMENTATION_BACKEND)
```

**Почему это работает:** клиент не знает, откуда пришёл STL. Аппликация работает одинаково. Переключение — одна строка в `.env`.

## 4. МОДУЛЬ 3 — ML

### 4.1 Назначение
Всё, что связано с **сегментацией органов** и **расчётом троакаров**. Модуль работает как на стороне сервера (генерация STL), так и частично на клиенте (предсказание точек троакаров на основе уже загруженной модели).

### 4.2 Стек и библиотеки

| Компонент | Решение | Примечание |
|---|---|---|
| Быстрая сегментация (now) | TotalSegmentator | Работает без GPU, установка pip, хорошее качество |
| Сегментация (future) | nnU-Net | Fine-tune на KiTS21 + ваших данных. SOTA |
| DICOM → NIfTI | SimpleITK | Обязательная конверсия перед ML |
| Маски → STL | scikit-image (`marching_cubes`) | |
| Обработка таблиц смещения | pandas + openpyxl | Парсинг Excel → JSON (displacement vectors) |
| Визуальная проверка | 3D Slicer | Не код — инструмент для валидации сегментации |
| Расчёт троакаров | Custom Python | Геометрия: проекция на брюшную стенку, углы, глубина |
| Future: displacement prediction | PyTorch (custom model) | Multi-task: сегментация + предсказание смещения |

### 4.3 Пайплайн ML (текущий и целевой)

```
ТЕКУЩИЙ (Test, быстрый старт):
  DICOM → SimpleITK → NIfTI → TotalSegmentator → masks → marching_cubes → STL

ЦЕЛЕВОЙ (Prod, через 1–2 недели):
  DICOM → SimpleITK → NIfTI → nnU-Net (KiTS fine-tune) → masks → STL
                                        ↑
                              displacement tables (Excel) → JSON metadata

ТРОАКАРЫ (отдельный модуль):
  STL почки + metadata (ИМТ, смещение) → расчёт оптимальных точек → JSON
```

### 4.4 Структура ML-модуля

```

│   ├── create_nnunet_dataset.py       // подготовка датасета для nnU-Net
│   └── train_nnunet.py                // запуск обучения (future)
├── inference/

│   └── stl_exporter.py                // masks → binary STL
```

## 5. МОДУЛЬ 4 — AR (внутри Unity-клиента)

### 5.1 Назначение
Подсистема регистрации и визуализации. Этот модуль часто путается с "клиентом" в целом, но это отдельная ответственность: всё что связано с пространственным совмещением модели с реальным миром.

### 5.2 Стек

| Компонент | Решение |
|---|---|
| Трекинг сцены | ARCore (через ARFoundation) |
| Маркеры | ArUco (OpenCV for Unity) |
| Регистрация по костям | Custom: SolvePnP (OpenCV) на 3 landmarks |
| Fusioning | Weighted average: маркеры (высокий вес) + кости (fallback) |
| Визуализация | Unity Mesh Renderer + custom Shader (полупрозрачность, LOD) |

### 5.3 Схема регистрации

```
Камера → OpenCV:
  ├── ArUco detect → pose маркера (если видно)
  └── Bone landmarks detect → 3 точки (12 ребро, подвздошная, остистые)
         ↓
  CombinedRegistration:
    ├── Если маркеры + кости → fusioning (вес 60/40)
    ├── Если только кости   → SolvePnP (3 точки → rigid transform)
    └── Если ничего         → FB-3: предупреждение, стоп
         ↓
  Confidence score (0–1):
    ├── ≥ 0.85 → зелёный, работаем
    ├── 0.7–0.85 → жёлтый, предупреждение
    └── < 0.7  → красный, перерегистрация
         ↓
  User verification: "Подтвердить" / "Перерегистрация"
```

## 8. ОЦЕНКА И КОММЕНТАРИИ

### 8.1 Что сделано хорошо
Документация проекта (PRD, ТЗ) — на высоком уровне. Fallback-сценарии описаны детально, что редко встречается на стадии MVP. DICOM-пайплайн уже частично реализован и тестирован на реальных данных — это существенная фора.

### 8.2 Критичные моменты, требующие внимания

**1. Test API — это не опционально, это блокер.**  
Без возможности получить STL без Амиры весь клиент (Unity) нельзя развивать. TotalSegmentator или pre-built mock — нужно одно из двух *сейчас*, до старта спринта 2. Рекомендация: начать с pre-built STL из реальных данных (они уже есть у вас), а TotalSegmentator запустить параллельно как более автоматизированный вариант.

**2. React-прототип — это только для визуальной референсы.**  
React-макет полезен, чтобы согласовать UX с заказчиком. Но вся боевая логика — в Unity. Не стоит пытаться его "интегрировать" — просто используйте как скриншоты/макеты при создании Unity UI.

**3. ML и Сервер — тесно связаны, но их можно развивать параллельно.**  
ML-модуль (`TotalSegmentator` → STL) может работать offline как скрипт. Сервер берёт его результаты и раздаёт через API. Связка не требует одновременной готовности — достаточно договориться о формате STL + JSON (он уже описан в спецификации STL-1–STL-6).

**4. Задача "Multi-organ сегментация" (бывшая задача 7 в TASKS) сформулирована слишком широко.**  
Печень и селезёнка нужны *только* для определения стороны почки (FR-0.2). Это не полноценная multi-organ сегментация — это один шаг валидации. Переформулировано в ML-4 соответственно.

**5. Задачи 12, 13, 16, 17 из старого TASKS — это Post-MVP.**  
Email-уведомления, PDF-отчёты, PACS-интеграция, iOS-приложение — ни одно из этих не входит в 6-недельный MVP по спецификации. Они оставлены для информации, но не включены в основной plan.

**6. Точность регистрации ≤ 5 мм — самое критичное требование.**  
Всё, что влияет на это (выбор ландмарков, качество SolvePnP, фуссион с маркерами) — приоритет 1. Если в спринте 3 точность не достигается — нужно менять стратегию (больше ландмарков, другие маркеры, калибровка камеры).

### 8.3 Предложения по оптимизации процесса

Рекомендуется ввести **единую mock-среду** (`mock_data/`), в которую кладётся всё:
- DICOM-серии (реальные)
- Pre-built STL (из 3D Slicer или TotalSegmentator)
- Маркерные фото/видео (для тестирования трекинга без физического планшета)
- poses.json (для Unity PC Preview)
- displacement_metadata.json (из таблиц)

Это позволит каждому модулю тестироваться автономно, без зависимости от live-сервера и реального оборудования.
