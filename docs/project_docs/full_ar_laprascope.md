## 1) Итоговая цель системы (как должно работать)

### 1.1 Пользовательский поток (целевой)
1) Оператор на планшете (Unity) выбирает DICOM ZIP (или NIfTI)
2) Unity отправляет файл на сервер (`POST /api/v1/upload` или `/upload-nifti`)
3) Сервер:
   - валидирует вход
   - выполняет пайплайн (сегментация → NIfTI → STL)
   - формирует `metadata.json`, включая `displacement` (точки привязки + векторы в мм)
4) Unity:
   - опрашивает статус (`GET /api/v1/status/{job_id}`)
   - скачивает STL/NIfTI/metadata
   - визуализирует 3D + смещения (вектора/подсказки)
5) Сервер автоматически удаляет временные данные и, по TTL, результаты.

### 1.2 Ключевые артефакты, которые должны существовать
- **STL**: минимум `kidney_left.stl`, `kidney_right.stl` (или один орган для MVP)
- **Metadata JSON**: status + displacement + технические параметры (оси, масштаб, units)
- **Downloads**: возможность скачать всё одним архивом (для отладки)

## 3) Архитектурные решения (зафиксировать)

### 3.1 Что и когда удаляем (агрессивная очистка + TTL=1 день)
Из-за нехватки диска: **временные данные чистим сразу**, результаты — по TTL.

- **Сразу после успешной генерации STL (post-success cleanup):**
  - удалить распакованный `dicom/`
  - удалить промежуточные NIfTI (Unity для MVP достаточно `STL + metadata`, без NIfTI)
  - оставить минимально необходимые файлы:
    - STL
    - metadata.json
    - status.json

- **TTL очистка результатов (hard TTL):**
  - TTL по умолчанию: **24 часа**
  - после TTL удалить job полностью

- **Важное ограничение:** Unity должна успеть скачать результаты до TTL.

### 3.2 Контракт данных для смещения (минимум на первом этапе)
Смещение нужно отдавать в виде:
- **anchors** (точки привязки, координаты)
- **vectors** (dx/dy/dz, мм)
- **координатная система и смысл осей**

Минимальный набор anchors, который ты хочешь (зафиксировано):
- **полюса** почки (верхний/нижний)
- **ворота** почки (hilum)

То есть для каждой почки на MVP: 3 точки (upper pole, lower pole, hilum) + **вектор смещения для каждой точки**.

Рекомендуемый JSON (можно расширять):
```json
{
  "displacement": {
    "units": "mm",
    "coordinate_system": "LPS|RAS|custom",
    "frame": "patient",
    "anchors": {
      "kidney_left": {
        "upper_pole": {"x": 0, "y": 0, "z": 0},
        "lower_pole": {"x": 0, "y": 0, "z": 0},
        "hilum": {"x": 0, "y": 0, "z": 0}
      },
      "kidney_right": {
        "upper_pole": {"x": 0, "y": 0, "z": 0},
        "lower_pole": {"x": 0, "y": 0, "z": 0},
        "hilum": {"x": 0, "y": 0, "z": 0}
      }
    },
    "vectors": {
      "kidney_left": {
        "upper_pole": {"dx": 0, "dy": 0, "dz": 0},
        "lower_pole": {"dx": 0, "dy": 0, "dz": 0},
        "hilum": {"dx": 0, "dy": 0, "dz": 0}
      },
      "kidney_right": {
        "upper_pole": {"dx": 0, "dy": 0, "dz": 0},
        "lower_pole": {"dx": 0, "dy": 0, "dz": 0},
        "hilum": {"dx": 0, "dy": 0, "dz": 0}
      }
    },
    "notes": {
      "dx_meaning": "left(-)/right(+)",
      "dy_meaning": "posterior(-)/anterior(+)",
      "dz_meaning": "inferior(-)/superior(+)"
    }
  }
}
```

Примечание: если в будущем понадобятся дополнительные точки (например «центр», точки вдоль оси, сосудистые ориентиры), добавляем их как новые ключи в `anchors.*` и `vectors.*` без ломки обратной совместимости.

### 3.3 Где выдавать displacement

## 5) Риски и как их закрывать

## 5.1) Приоритизация: что делаем строго по MVP, а что откладываем

### MVP (делаем сейчас)
- Unity: upload → polling → download STL+metadata → простая визуализация
- Backend: стабильный запуск + таймауты + агрессивная cleanup + TTL 24h
- Displacement: anchors (полюса+ворота) + per-anchor vectors + единицы/оси

### После MVP (делаем только когда выше стабильно)
- ARFoundation регистрация/трекинг
- Сложная визуализация (анимации деформаций/сосудов)
- ML модель displacement (после data QA)
- База данных пациентов/версийность (если потребуется)

- **Долгая обработка CPU**:
  - Nginx timeouts
  - статус прогресса

- **Неоднозначность осей displacement**:
  - фиксировать `coordinate_system` + `notes` в JSON
